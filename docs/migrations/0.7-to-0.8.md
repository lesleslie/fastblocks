# Migration Guide: 0.7 → 0.8

This release removes the dangerous MCP surface that has been moved to consumer
applications, completes the ACB → Oneiric migration, swaps `exec_module` for
`ast.parse` in the HTMY component loader, and switches the dependency
declaration from `>=` to `~=` (compatible release). Most of these changes are
mechanical; the breaking items are listed below with a one-line "how to
migrate" for each.

## 1. Resolver: ACB → Oneiric

The four integration modules (`_events_integration`, `_health_integration`,
`_validation_integration`, `_workflows_integration`) used to each instantiate
their own `oneiric.core.resolution.Resolver`. They now share a single
process-wide singleton via `fastblocks.core.resolver.get_resolver()`.

### If your code imported from the integration modules

No change. The public function signatures are unchanged.

### If your code did this:

```python
from oneiric.core.resolution import Resolver

depends = Resolver()
```

### Change to:

```python
from fastblocks.core.resolver import get_resolver

depends = get_resolver()
```

`get_resolver()` is lazy-initialised, so import-time side effects are not paid
at module load. The returned object is a process-wide singleton; do not cache
a copy in a module-level constant.

### If your code used ACB `Depends` / `Inject` / `depends.get`

The `acb` extra has been removed. The new pattern is the typed Python API on
each adapter (e.g. `AdapterRegistry.configure(adapter_name, **fields)`) or a
direct call to `get_resolver().resolve("fastblocks", "<key>")`. There is no
direct ACB replacement; the Oneiric resolver is the canonical dependency
injection surface from 0.8.0 forward.

## 2. HTMY component loader: `exec_module` → `ast.parse`

`_load_component_from_source` in
`fastblocks/adapters/templates/_htmy_components.py` no longer executes source
code via `importlib.util.spec_from_file_location + exec_module`. It now parses
the source with `ast.parse` and walks for class / function definitions only.

### What is now rejected

- Top-level `Import` / `ImportFrom` statements
- Calls to `exec` / `eval` / `compile` / `__import__`
- Top-level statements other than class / function definitions

### If you shipped a component that relied on import-time side effects

Refactor those side effects into the class body (`__post_init__`,
`__init_subclass__`, a classmethod that runs once via `functools.cache`, etc.).
The framework calls the `htmy(context)` method on your class; that's the only
contract you need to keep.

## 3. Deleted MCP tools (slated for 0.8.0)

The following MCP tools / modules are removed in 0.8.0:

- All of `fastblocks.mcp.websocket_tools` (7 tools: `fastblocks_start_websocket`,
  `fastblocks_stop_websocket`, `fastblocks_websocket_status`,
  `fastblocks_broadcast_ui`, `fastblocks_broadcast_state`,
  `fastblocks_broadcast_component`, `fastblocks_list_subscriptions`)
- `fastblocks.mcp.config_cli` (the Click wizard)
- The `create_template` MCP wrapper in `fastblocks.mcp.tools`
- The `create_component` MCP wrapper in `fastblocks.mcp.tools`
- The `configure_adapter` MCP wrapper in `fastblocks.mcp.tools`
- The fake `get_route_definitions` resource in `fastblocks.mcp.resources`
- The unused `register_fastblocks_tools_async` in `fastblocks.mcp.tools`

### How to migrate

These are product operations, not framework introspection. The right place for
them is the consumer app's MCP server. If you are building such a server, see
the (forthcoming) "SplashStand MCP rebuild" plan for the typed
`splashstand_*` tool surface.

In the meantime, the **Python API** for every removed tool remains available:

- `from fastblocks.websocket import FastblocksWebSocketServer` and call
  `await server.start()` directly from your consumer app.
- `from fastblocks.adapters.templates._htmy_components import scaffold_component`
  for the create-component path.
- `from fastblocks.mcp.registry import AdapterRegistry` and call
  `await registry.configure(adapter_name, **fields)` for the adapter-config
  path.

The CLI commands (`fastblocks create app`, `fastblocks create template`,
`fastblocks create ide-config`) are also unaffected.

## 4. Dependency pins: `>=` → `~=`

The 18 runtime deps are now declared with `~=` (compatible release). For
example, `pydantic>=2.12.4` became `pydantic~=2.12`, meaning
`>=2.12,<2.13`. This matches the crackerjack rule for stable dependencies.

**For library consumers:** nothing changes at the install level. `uv add
fastblocks` resolves the same versions it did before, modulo the major-version
ceiling now enforced.

**For application authors who pinned transitive deps:** verify your pin does
not conflict with the new ceiling. If you have `pydantic>=2.13` in your
application's `pyproject.toml`, the resolver will choose between your pin and
the fastblocks pin; widen your range or remove it.

The one exception is `sentry-sdk[starlette]`, which is kept as
`>=3.0.0a7` because its only published 3.x release is a prerelease.

## 5. `acb` extra removed

The `[dependency-groups].cache` group is gone. If you had:

```toml
[tool.uv.sources]
fastblocks = { index = "...", extra = "cache" }
```

…or you were installing fastblocks with `pip install fastblocks[cache]`, that
extra no longer exists. The new installation is just `pip install fastblocks`;
there is no replacement for the cache-specific configuration that the `acb`
extra provided.

## 6. `session-buddy` dropped from dev extras

The dev dependency `session-buddy>=0.9.4` was declared in 0.7.x but had zero
importers in the fastblocks tree. It is gone. If you were relying on
session-buddy being pulled in transitively by the fastblocks dev install,
declare it explicitly in your application's `pyproject.toml`.

## 7. Logger import surface

If your application code imported `fastblocks.X.Y` and used the stdlib
`logging.getLogger` re-exported by those modules, that re-export is gone.
Switch to `oneiric.core.logging.get_logger(__name__)`. The new convention is
uniform across `fastblocks/websocket/`, `fastblocks/mcp/`, and
`fastblocks/shell/`.

## 8. WebSocket env vars: still read at import time (work in progress)

**This is a known not-yet-landed item.** `fastblocks/websocket/auth.py` still
reads `FASTBLOCKS_JWT_SECRET`, `FASTBLOCKS_TOKEN_EXPIRY`, and
`FASTBLOCKS_AUTH_ENABLED` at **module load time**. The 0.8.0 plan calls for
this to move to use-time reads, with `JWT_SECRET ==
"dev-secret-change-in-production"` hard-failing in non-test processes. If you
are consuming the 0.8.0 release and the hard-fail has landed, you will need to
set `FASTBLOCKS_JWT_SECRET` before `import fastblocks.websocket` runs. If the
hard-fail has not yet landed, you will see a warning but the server will
start. Either way, **set `FASTBLOCKS_JWT_SECRET` in your production
environment.**

## 9. Pre-existing test failures

The release notes list 19 collection errors and ~80 failing tests under
`pytest -m "not slow"`. Most are pre-existing test-infra issues (xdist
`sys.modules` pollution in conftest, mock signature drift, etc.) and are not
introduced by this release. See `docs/ONEIRIC_DEPENDS_PATTERNS.md` and the
[Pre-existing failure brief] for the per-cluster diagnosis. If your
application's tests are affected, the workaround is `pytest -p no:xdist` for
the affected file.
